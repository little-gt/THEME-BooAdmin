# 🔐 安全策略 (Security Policy)

感谢您对 **BooAdmin** 的关注！我们致力于为 Typecho 用户提供既美观又安全的后台环境。虽然这是一个由社区维护的开源项目，但我们非常重视每一个安全反馈。

---

## 📌 支持的版本

为了确保资源能集中在最新的代码库上，我们目前仅对以下版本提供安全支持：

| 版本 | 状态 | 说明 |
| :--- | :--- | :--- |
| **1.1.2** | ✅ **最新正式版** | 基于 TailwindCSS 重构，完全适配 Typecho 1.3.0+，新增移动端自适应和视图切换功能，提供当前最优先的安全维护。 |
| **1.0.2** | ⚠️ **仅限安全更新** | 传统的 Bootstrap 版本，仅针对严重安全漏洞提供维护，建议尽快迁移至 v1.1.2。且仅支持到1.1.1发布后60天。 |
| **< 1.0.2** | ❌ **已停止支持** | 不再提供任何更新。 |

## 📮 报告安全漏洞

如果您在 BooAdmin 中发现了潜在的安全漏洞，请**不要**在 GitHub Issues 或其他公共论坛中直接披露。这有助于防止漏洞在修复发布前被恶意利用。

请遵循以下步骤进行**负责任的披露**：

### 1. 联系方式
请将漏洞详细信息发送至安全联系邮箱：
📧 **[coolerxde@gt.ac.cn](mailto:coolerxde@gt.ac.cn)**

### 2. 邮件格式
为了帮助我们要快速分类和处理，请在邮件标题中使用以下格式：
`[Security] BooAdmin - <漏洞简述>`

### 3. 报告内容
为了加快修复速度，请尽量包含以下信息：

*   **影响版本**：例如 v1.1.2，或说明是否影响所有版本。
*   **漏洞类型**：如 XSS, CSRF, SQL 注入, 文件上传漏洞, 越权访问等。
*   **影响范围**：说明漏洞可能造成的具体危害（如数据泄露、权限提升等）。
*   **复现步骤**：清晰的文字描述或复现视频，确保我们能够重现问题。
*   **PoC (概念验证)**：如果可能，提供一段最小化的复现代码或 HTTP 请求包。
*   **受影响环境**：PHP 版本、Typecho 版本、浏览器类型等（如相关）。

---

## 📢 响应与披露流程

1.  **确认收到**：我们承诺在收到报告后的 **48 小时内** 给予回复确认。
2.  **评估修复**：我们将评估漏洞的严重程度，并着手开发修复补丁。在此期间，请您保持静默，不要公开披露。
3.  **发布更新**：修复完成后，我们将发布新版本（如 v1.1.x），并在更新日志中注明安全修复内容。
4.  **公开致谢**：在漏洞修复并发布后，如果你愿意，我们将在安全公告或 Release Note 中对您的贡献进行致谢。请明确告知是否希望在致谢中提及你的姓名或 ID。

## 📦 安全范围

本策略仅涵盖 **BooAdmin 主题本身** 的代码：

*   ✅ `admin/` 目录下由本主题修改或新增的 PHP 文件。
*   ✅ `admin/js/` 和 `admin/css/` 目录下的主题资源文件。
*   ✅ 主题的逻辑漏洞、权限控制问题、XSS/CSRF 等前端安全问题。

**以下情况不在本项目的安全处理范围内：**

*   ❌ **Typecho 核心** 自身的漏洞（请直接向 [Typecho 官方仓库](https://github.com/typecho/typecho) 报告）。
*   ❌ **第三方插件** 引入的安全问题（请联系相应插件作者）。
*   ❌ 服务器环境配置不当（如 PHP 版本过低、目录权限设置错误、开放端口过多等）。
*   ❌ 针对管理员账号的暴力破解或社交工程攻击。
*   ❌ 与 BooAdmin 无关的系统级漏洞。

## 🛡️ 安全最佳实践

为了提高您的后台安全性，我们建议您：

1.  **保持更新**：始终使用最新版本的 Typecho 和 BooAdmin 主题。
2.  **强密码策略**：使用复杂密码，并定期更换管理员密码。
3.  **HTTPS 启用**：为您的网站配置 SSL 证书，启用 HTTPS 加密传输。
4.  **定期备份**：定期备份数据库和文件，以便在发生意外时快速恢复。
5.  **限制访问**：如果可能，限制 `/admin` 目录的访问 IP，或使用额外的登录验证（如云锁、宝塔面板的登录保护）。
6.  **审核插件**：仅从可信来源安装插件，并定期检查插件更新。

## 📄 安全报告示例

**主题**: `[Security] BooAdmin - 存储型 XSS 漏洞`

**正文**:
> 您好，我在 v1.1.2 版本中发现了存储型 XSS 漏洞。
>
> **影响版本**: v1.1.2
>
> **漏洞描述**: 在撰写文章时，如果评论内容包含恶意脚本，会被存储并执行。
>
> **复现步骤**:
> 1. 登录后台
> 2. 新建评论，内容为 `<script>alert(1)</script>`
> 3. 访问对应文章，脚本被执行
>
> **PoC**: `<img src=x onerror=alert('XSS')>`
>
> **影响**: 攻击者可以窃取管理员 Cookie，获得后台访问权限。
>
> 请确认并尽快修复，我同意在修复后公开致谢。
